
# Report: 다중 창(grid 배치) vs 단일 창(canvas 합성) 비교 분석

대상 코드: Report02/TEST.py

이 리포트는 **기존 방식(여러 개의 OpenCV 윈도우를 띄워 격자 배치)**과, **수정된 방식(하나의 큰 캔버스 이미지로 합성 후 단일 윈도우에 출력, WSL에서 자동 적용)**을 비교한다.

---

## 1) 기존 방식: 다중 창(윈도우 여러 개) 격자 배치

### 핵심 구현 아이디어
- 이미지 5장을 각각 별도의 윈도우로 띄운다.
- 각 윈도우에 대해 다음을 수행한다.
	- `cv.namedWindow(title, cv.WINDOW_NORMAL)`
	- `cv.imshow(title, img)`
	- `cv.resizeWindow(title, w, h)`
	- `cv.moveWindow(title, x, y)`
- 즉, “격자 화면”을 만들기 위해 **윈도우 매니저(운영체제 GUI 시스템)의 창 배치 기능**에 의존한다.

### 장점
- 구현이 직관적이다(각 이미지는 각 창에 바로 보여준다).
- 각 창을 독립적으로 확대/이동하면서 관찰할 수 있다.

### 단점 / 환경 의존성
- `moveWindow`, `resizeWindow`는 **요청(request)**에 가까워서 운영체제/윈도우 매니저가 완전히 동일하게 반영해 주지 않을 수 있다.
- Windows에서는 비교적 의도대로 배치되는 경우가 많지만, WSL 환경에서는 다음 요인 때문에 결과가 달라질 수 있다.
	- WSLg(WSL2): Linux GUI(wayland/X11) → RDP 중계 과정에서 창 정책이 달라짐
	- X 서버 사용: X 서버/윈도우 매니저 설정, DPI 스케일링 차이
	- 멀티 모니터/배율(DPI) 설정 차이로 절대 좌표가 기대와 달라짐
- 창이 여러 개라서 포커스/입력(`waitKey`) 처리도 사용 환경에 따라 체감이 달라질 수 있다.

---

## 2) 수정된 방식: 단일 창(캔버스 합성) 그리드 출력 + WSL 자동 분기

### 핵심 구현 아이디어
- 여러 이미지를 각각 창으로 띄우지 않고,
	1) 큰 캔버스(`np.zeros`)를 만들고
	2) 각 이미지를 타일 크기로 리사이즈한 뒤(`cv.resize`)
	3) 캔버스의 해당 위치에 복사하여 “그리드 1장”을 구성한다.
- 이후 **창은 1개만 띄워** `imshow(canvas)`로 출력한다.

### 추가된 보조 처리
- `_to_bgr(img)`:
	- grayscale(2D) 이미지(Y 채널)는 `cv.cvtColor(..., COLOR_GRAY2BGR)`로 3채널로 변환
	- 그래야 캔버스(BGR 3채널)에 자연스럽게 합성 가능
- 타이틀 표기:
	- `cv.putText`로 각 타일 영역에 문자열 타이틀을 얹는다.

### WSL 자동 분기
- `_is_wsl()`로 WSL 환경을 감지하고,
	- WSL이면: `show_imgs_by_grid_single_window(...)`
	- 아니면: 기존 `show_imgs_by_grid(...)`
	로 동작을 분기한다.

### 장점
- OS/WSL/윈도우 매니저 차이에 덜 민감하다.
	- “창 배치”가 아니라 “이미지 합성”으로 격자 모양을 만들기 때문
- 결과가 일관된다(특히 WSLg에서 창 위치가 제멋대로 되는 문제를 회피).
- 창이 하나라서 사용자 입장에서 관리(닫기/포커스)가 단순하다.

### 단점
- 각 타일이 `tile_size`로 강제 리사이즈되므로, 원본 해상도 그대로 1:1 관찰은 어려워질 수 있다.
- 창을 독립적으로 확대해서 보는 “다중 창 방식의 장점”은 줄어든다.
- 텍스트(`putText`)가 이미지 위에 오버레이되므로, 픽셀 단위 분석이 목적이면 방해가 될 수 있다.

---

## 3) 두 방식의 차이 요약(표)

| 항목 | 기존(다중 창 배치) | 수정(단일 창 캔버스) |
|---|---|---|
| 격자 구성 방식 | 여러 윈도우를 좌표로 배치 | 한 장의 이미지로 합성 |
| OS/WSL 의존성 | 큼(윈도우 매니저 정책 영향) | 작음(합성 결과는 동일) |
| 조작 편의 | 창이 많아 관리 번거로움 | 창 1개로 단순 |
| 세부 관찰 | 창별 확대/이동 유리 | 타일 리사이즈로 제한 |
| 구현 복잡도 | 단순 | 합성/채널 처리 필요 |

## 4) 결론 및 권장안

- **WSL 환경에서 “윈도우 격자 배치가 Windows와 다르게 보이는 문제”**를 해결하려면, 단일 창 캔버스 합성 방식이 가장 확실하다.
- Windows에서 창을 개별로 확대/비교하는 사용성이 중요하다면, 현재처럼 **WSL에서는 단일 창**, **Windows에서는 다중 창**을 사용하는 분기 방식이 합리적이다.

## 5) 코드 수정 사항(무엇을 어떻게 바꿨는가)

이 섹션은 “동작 변화”가 발생한 지점을 기준으로 실제 코드 변경 내용을 요약한다.

### (A) 이미지 로딩 경로 수정 + 실패 시 예외 처리 추가

**문제(수정 전)**
- 상대 경로 문자열로 이미지를 로드했는데(`./Report2/window.jpg`), 폴더명이 실제와 다르거나(Report02 vs Report2), 실행 위치(cwd)에 따라 경로가 달라져 `cv.imread`가 실패할 수 있었다.
- `imread` 실패 시 `img=None`인데, 이후 `cv.cvtColor(img, ...)`에서 OpenCV assertion(`!_src.empty()`)으로 크래시가 발생했다.

**수정(수정 후)**
- 스크립트 파일 위치 기준으로 경로를 계산하도록 변경했다.
	- `Path(__file__).resolve().parent / 'window.jpg'`
- 이미지 로드 실패 시 즉시 `FileNotFoundError`를 발생시켜 원인을 명확히 했다.
	- 결과적으로 “왜 안 뜨는지”가 OpenCV 내부 에러가 아니라 파일 경로 문제로 바로 드러난다.

### (B) WSL 환경 감지 함수 추가

**추가된 코드**
- `_is_wsl()` 함수를 추가하여 WSL 환경인지 여부를 판단한다.
- 감지 기준은 환경 변수(예: `WSL_DISTRO_NAME`, `WSL_INTEROP`) 또는 커널 릴리즈 문자열에 `microsoft` 포함 여부다.

**의도**
- WSL에서만 창 배치가 달라지는 현상을 회피하기 위해, 표시 방식(display strategy)을 자동으로 분기한다.

### (C) 단일 창 그리드 출력 함수 추가(캔버스 합성)

**추가된 코드**
- `show_imgs_by_grid_single_window(...)`를 새로 추가했다.
- 동작은 다음 순서다.
	1) `canvas = np.zeros((rows * tile_h, cols * tile_w, 3), dtype=np.uint8)`로 큰 이미지(캔버스) 생성
	2) 각 입력 이미지를 타일 크기로 `cv.resize`하여 크기 통일
	3) 타일을 캔버스의 지정 위치에 복사하여 그리드 합성
	4) `cv.putText`로 타일마다 타이틀 문자열 표시
	5) 최종 합성된 `canvas`를 `imshow`로 1개 창에 표시

**보조 함수 추가 이유**
- `_to_bgr(img)`:
	- 원본 BGR은 그대로 사용
	- grayscale(2D) 이미지(Y709)는 3채널(BGR)로 변환 후 합성
	- 그렇지 않으면 3채널 캔버스에 2D 배열을 넣을 때 shape mismatch가 발생한다.

### (D) 출력부(call-site) 수정: WSL이면 단일 창, 아니면 기존 방식 유지

**수정 전(개념적으로)**
- 항상 `show_imgs_by_grid(...)`만 호출하여 다중 창을 띄움

**수정 후**
- 출력 직전에 `imgs`, `titles` 리스트를 만들고,
- `if _is_wsl(): ... else: ...`로 표시 함수를 선택한다.
	- WSL: `show_imgs_by_grid_single_window(...)`
	- 기타: `show_imgs_by_grid(...)`

### (E) 사용자 체감 변화

- Windows:
	- 기존처럼 여러 창이 뜨고, 격자 배치 시도(`moveWindow/resizeWindow`)가 유지된다.
- WSL:
	- 여러 창 배치의 불확실성을 피하고, 1개의 창에서 동일한 그리드 레이아웃을 보게 된다.
- 공통:
	- 이미지 경로 문제로 인한 크래시가 사라지고, 경로가 잘못되면 즉시 예외로 원인이 드러난다.
